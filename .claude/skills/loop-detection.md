# loop-detection 스킬

> **용도**: Harness Engineering 원칙 3 — Loop Detection & Mitigation  
> **트리거**: sprint-dev 세션에서 루프 상태 조건 충족 시 자동 배정  
> **참조 규칙**: `.claude/rules/harness-engineering.md` 원칙 3

---

## 루프 상태 정의

아래 중 **하나라도** 해당하면 **루프 상태**로 판단하고 이 스킬을 즉시 실행한다:

| 조건 | 기준 |
|------|------|
| **테스트 루프** | 동일한 오류 메시지로 테스트가 **3회 이상 연속 실패** |
| **파일 수정 루프** | 동일한 파일을 **3회 이상 반복 수정** (동일 Task 내) |

---

## 루프 감지 시 즉시 처리 절차

### Step 1: 작업 즉시 중단

추가 코드 수정을 **완전히 멈춘다**. 다음 수정이 성공할 것이라는 근거 없는 낙관으로 시도를 반복하지 않는다.

### Step 2: 현황 데이터 수집

아래 데이터를 수집하여 분석한다:

**테스트 루프의 경우:**
- 실패한 테스트 이름 (함수/클래스명)
- 반복된 오류 메시지 전문
- 각 시도에서 적용한 수정 방법
- 실패 직전 git log (`git log --oneline -5`)

**파일 수정 루프의 경우:**
- 반복 수정된 파일 경로
- 각 수정에서 변경한 내용 요약
- 수정 목적 (어떤 기능/버그를 해결하려 했는지)
- scope.md의 완료 기준과 비교

### Step 3: 근본 원인 분석

수집된 데이터를 기반으로 아래를 분석한다:

1. **패턴 파악**: 3번의 시도에서 공통점과 차이점은?
2. **실패 원인 가설**: 왜 동일한 접근법이 반복 실패하는가?
   - 잘못된 전제(wrong assumption)
   - 의존성/환경 문제
   - 아키텍처 불일치
   - 테스트 자체의 결함
3. **놓친 정보**: 읽지 않은 관련 파일이 있는가?

### Step 4: 보고 형식

사용자에게 아래 형식으로 보고한다:

```
🔁 루프 상태 감지

**유형**: [테스트 루프 / 파일 수정 루프]
**대상**: {파일명 또는 테스트명}
**반복 횟수**: {N}회

---

**시도 이력**:
1. {첫 번째 시도}: {적용한 수정} → 실패 이유: {오류}
2. {두 번째 시도}: {적용한 수정} → 실패 이유: {오류}
3. {세 번째 시도}: {적용한 수정} → 실패 이유: {오류}

**실패 원인 분석**:
{공통 패턴, 가설, 근본 원인 추정}

**추정 근본 원인**:
{가장 가능성 높은 원인}

**권장 조치** (사용자 판단 필요):
A) {접근법 A — 예: 아키텍처 재검토}
B) {접근법 B — 예: 관련 파일 추가 탐색}
C) {접근법 C — 예: 외부 도움/문서 참조}

→ 진행 방향을 결정해주시면 계속하겠습니다.
```

### Step 5: 사용자 승인 대기

보고 후 **사용자의 명시적 지시를 기다린다**. 지시 없이 재시도를 시작하지 않는다.

## 파일 수정 횟수 추적 방법

sprint-dev 세션에서 `docs/sprint/sprint{n}/scope.md`의 수정 파일 목록에 횟수를 기록한다:

```markdown
## 이번 세션에서 수정할 파일
| 파일 | 수정 횟수 | 비고 |
|------|---------|------|
| src-tauri/src/commands/auth.rs | [1회] | 로그인 로직 추가 |
| src-tauri/src/commands/auth.rs | [2회] | 버그 수정 |
| src-tauri/src/commands/auth.rs | [3회 ⚠️] | → loop-detection 실행 |
```

**3회 도달 시**: 수정 전에 loop-detection 스킬을 실행하고, 사용자 승인 후에만 수정을 계속한다.

## 루프 예방 체크리스트

루프에 빠지기 전에 아래를 확인한다:

- [ ] 실패 오류 메시지를 **완전히** 읽었는가? (첫 줄만 읽지 않기)
- [ ] 관련 파일(`import`되는 파일, 설정 파일)을 충분히 읽었는가?
- [ ] 수정 전에 해당 함수/모듈의 **전체 맥락**을 이해했는가?
- [ ] 테스트가 실제로 **올바른** 동작을 검증하는가? (테스트 자체 오류 가능성)
- [ ] 환경 변수, DB 상태 등 **외부 의존성** 문제는 없는가?
