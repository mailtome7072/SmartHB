# Systematic Debugging

버그의 근본 원인을 체계적으로 파악하고 수정하는 절차입니다.
`sprint-dev.md`에서 버그/오류 Task에 자동 배정됩니다.

## 5단계 절차

### 1단계: 증상 정의 (What)
- 어떤 동작이 일어나는가? (실제 동작)
- 어떤 동작이 기대되는가? (기대 동작)
- 언제, 어떤 조건에서 발생하는가? (재현 조건)
- 오류 메시지/스택 트레이스 전문 수집

### 2단계: 범위 파악 (Where)
- 관련 파일과 함수를 추적합니다: 오류 메시지 → 호출 스택 → 진입점
- 변경 이력 확인: `git log --oneline -10 -- <파일>` 으로 최근 변경 파악
- 영향 범위: 이 버그가 다른 기능에 영향을 주는가?

### 3단계: 가설 설정 (Why)
가능한 원인을 2~3개 나열합니다:
```
가설 A: {원인} → 확인 방법: {방법}
가설 B: {원인} → 확인 방법: {방법}
```
- 가장 단순한 가설부터 검증합니다 (Occam's Razor)

### 4단계: 최소 재현 (Reproduce)
- 버그를 재현하는 가장 작은 테스트 케이스를 작성합니다
- 환경 의존성(DB, 외부 API, 랜덤 값)을 제거합니다
- **재현 가능한 테스트 없이 코드를 수정하지 않습니다**

### 5단계: 수정 → 검증
- 한 번에 하나의 가설만 수정합니다 (여러 곳을 동시에 바꾸지 않음)
- 수정 후 반드시 검증:
  ```bash
  cargo test {관련_테스트명} --manifest-path src-tauri/Cargo.toml   # Rust 백엔드
  pnpm tsc --noEmit && pnpm lint                                     # 프론트엔드
  ```
- 수정이 다른 기능을 깨뜨리지 않았는지 전체 테스트 실행

## 자주 발생하는 버그 유형별 체크

| 유형 | 첫 확인 포인트 |
|------|--------------|
| API 오류 (4xx/5xx) | 요청 파라미터·헤더·인증 토큰 확인 |
| DB 오류 | 마이그레이션 누락, N+1 쿼리, 트랜잭션 누수 |
| 타입 오류 | TypeScript strict 모드로 타입 추론 확인 |
| 비동기 버그 | Promise reject 핸들링, race condition |
| 환경 변수 | `.env` 값 누락 또는 잘못된 값 |

## 금지 행동

- 원인 파악 없이 코드 수정 → 재발 가능성 높음
- 여러 곳을 동시에 수정 → 원인 특정 불가
- `console.log` 수백 개 추가 후 미삭제
- `--no-verify`로 커밋 후 "나중에 고치기"
