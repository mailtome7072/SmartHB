# Brainstorming — 설계 대안 비교 및 의사결정

설계 대안을 정량·정성 분석하여 팀이 추적 가능한 방식으로 결정을 내리는 3단계 절차입니다.
`sprint-dev.md`에서 "A vs B", "설계 대안 비교", "아키텍처 결정" Task에 자동 배정됩니다.

---

## 1단계: Weighted Decision Matrix (정량 비교)

비교 기준에 가중치를 부여하고 각 선택지를 1–5점으로 평가하여 총점을 계산합니다.

### 기본 비교 기준 및 가중치

| 기준 | 기본 가중치 | 설명 |
|------|------------|------|
| 성능 | 0.25 | 응답시간, 처리량, 리소스 효율 |
| 비용 | 0.20 | 구현 비용, 운영 비용, 라이선스 |
| 관리 복잡도 | 0.20 | 유지보수 난이도, 운영 부담 |
| 팀 경험 | 0.20 | 팀의 기술 친숙도, 학습 곡선 |
| 확장성 | 0.15 | 트래픽·데이터 증가 대응 능력 |

> 기준 추가/제거 또는 가중치 조정이 필요하면 논의 후 변경한다. 가중치 합계는 반드시 1.0.

### 점수표 작성 형식

```markdown
| 기준       | 가중치 | 선택지 A | A 점수 | 선택지 B | B 점수 |
|------------|--------|----------|--------|----------|--------|
| 성능       | 0.25   | {이유}   | {1-5}  | {이유}   | {1-5}  |
| 비용       | 0.20   | {이유}   | {1-5}  | {이유}   | {1-5}  |
| 관리 복잡도 | 0.20  | {이유}   | {1-5}  | {이유}   | {1-5}  |
| 팀 경험    | 0.20   | {이유}   | {1-5}  | {이유}   | {1-5}  |
| 확장성     | 0.15   | {이유}   | {1-5}  | {이유}   | {1-5}  |
| **총점**   |        |          | **{Σ(점수×가중치)}** | | **{Σ}** |
```

**점수 기준**: 5 = 매우 유리 / 3 = 보통 / 1 = 매우 불리

총점 차이가 0.3 미만이면 통계적으로 비슷한 수준으로 판단하고 2단계 정성 분석을 더 중요하게 반영한다.

## 2단계: SWOT + Trade-off 분석 (정성 비교)

1단계 점수에 담기지 않는 맥락적 요소를 분석합니다.

### SWOT 형식

각 선택지에 대해 항목당 2–3개씩 작성합니다:

```markdown
**선택지 A: {이름}**
- Strengths: {강점 2-3개}
- Weaknesses: {약점 2-3개}
- Opportunities: {기회 2-3개}
- Threats: {위험 2-3개}

**선택지 B: {이름}**
- Strengths: ...
- Weaknesses: ...
- Opportunities: ...
- Threats: ...
```

### Trade-off 명시

"A를 선택하면 X가 좋아지지만 Y가 나빠진다"는 형식으로 명확하게 표현합니다:

```markdown
| 선택 시 | 개선 (↑) | 저하 (↓) |
|---------|----------|----------|
| A 선택  | {항목}   | {항목}   |
| B 선택  | {항목}   | {항목}   |
```

### Risk 식별

구현 또는 운영 중 발생 가능한 위험 요소를 작성합니다:

```markdown
| 리스크 | 관련 선택지 | 영향도 | 완화 방법 |
|--------|------------|--------|----------|
| {위험} | A / B / 공통 | 높음/중간/낮음 | {방법} |
```

## 3단계: ADR 문서화

1–2단계 결과를 바탕으로 결정을 내리고 팀이 추적할 수 있도록 ADR(Architecture Decision Record)을 작성합니다.

### ADR 번호 결정

```bash
ls docs/arch/adr-*.md 2>/dev/null | wc -l
# 결과값 + 1 이 새 ADR 번호 (없으면 001부터 시작)
```

### ADR 저장 경로

```
docs/arch/adr-{NNN}-{주제-kebab-case}.md
예: docs/arch/adr-001-cache-strategy.md
    docs/arch/adr-002-auth-provider.md
```

### ADR 파일 형식

```markdown
# ADR-{NNN}: {결정 제목}

- **상태**: Proposed  ← 팀 검토 완료 후 Accepted로 수동 변경
- **날짜**: {YYYY-MM-DD}
- **결정자**: {이름 또는 팀}

## Context (배경)

{왜 이 결정이 필요한가? 기술적·비즈니스적 제약 조건 설명}

## Decision (결정)

{선택한 대안과 핵심 이유 — 1단계 총점 및 2단계 핵심 Trade-off 요약}

> 1단계 총점: A = {X}, B = {Y} → {선택지} 채택
> 핵심 Trade-off: {A를 선택함으로써 얻은 것과 포기한 것}

## Consequences (영향)

**긍정적 영향:**
- {항목}

**부정적 영향 / 주의사항:**
- {항목}

**후속 액션 (있는 경우):**
- {항목}
```

> **Status 값**: `Proposed` (작성 직후) → `Accepted` (팀 확정 후 수동 업데이트) → `Superseded by ADR-{NNN}` (대체된 경우)

---

## 사용 흐름 요약

1. 비교할 선택지를 2–3개로 좁힌다.
2. 1단계 Weighted Matrix로 정량 점수를 산출한다.
3. 2단계 SWOT + Trade-off로 점수에 담기지 않는 맥락을 보완한다.
4. 두 결과를 종합해 선택지를 결정하고 3단계 ADR을 작성한다.
5. ADR 상태를 `Proposed`로 설정 — 팀 검토 후 `Accepted`로 수동 변경한다.

## 금지 행동

- 1단계 점수만 보고 결정 — Trade-off 맥락 없이 총점 높은 쪽을 자동 채택하지 않는다.
- ADR 생략 — 설계 결정은 반드시 `docs/arch/`에 기록한다.
- 선택지를 1개만 분석 — 최소 2개 이상 비교해야 의미 있는 매트릭스가 성립한다.
