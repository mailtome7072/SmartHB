# karpathy-guidelines

Andrej Karpathy가 제안하는 소프트웨어 개발 원칙을 이 프로젝트에 맞게 정리한 지침입니다.

## 적용 시점

이 지침은 **코드를 직접 작성하거나 수정하는 모든 시점**에 적용된다. (개발자 및 AI 모두 해당)

| 시점 | 적용 원칙 |
|------|-----------|
| 파일 수정 전 | 원칙 2 — 읽고 이해한 뒤 수정 |
| 기능 구현 시 | 원칙 1 — MVP 우선, 원칙 4 — 단순 유지 |
| 커밋 전 | 원칙 3 — 테스트 작성, 원칙 6 — 문서 동기화, 원칙 8 — 보안 확인 |
| CI 실패 시 | 원칙 7 — 원인 파악 및 수정 (`--no-verify` 우회 금지) |
| 브랜치 생성·PR 전 | 원칙 5 — 브랜치 전략 준수 |

## 핵심 원칙

### 1. 작게 시작하고, 점진적으로 확장한다
- 한 번에 많은 기능을 추가하지 말고, 작동하는 최소 단위부터 구현한다.
- 각 단계에서 코드가 실제로 동작하는지 검증한 뒤 다음 단계로 넘어간다.
- MVP(최소 기능 제품) 우선, 완성도는 이후 스프린트에서 높인다.

### 2. 코드를 직접 읽고 이해한 뒤 수정한다
- 파일을 수정하기 전에 반드시 현재 상태를 읽고 파악한다.
- 기존 패턴과 컨벤션을 유지한다.
- 수정 범위를 최소화하여 의도치 않은 사이드 이펙트를 예방한다.

### 3. 테스트를 작성하고 자동화한다
- 새 기능에는 반드시 pytest 테스트를 작성한다.
- CI에서 자동으로 테스트가 실행되도록 구성한다.
- 테스트가 없으면 기능이 완성된 것이 아니다.

### 4. 복잡성을 피하고, 단순하게 유지한다
- 추상화는 필요한 시점에만 추가한다.
- 3개 이상의 파일에 걸친 변경은 Sprint로 분류한다.
- "나중에 리팩토링하자"보다 처음부터 단순하게 짠다.

### 5. 브랜치 전략을 엄격히 준수한다
- `sprint{n}` 브랜치에서 개발 → `develop` PR → `main` 배포
- `main` 직접 push 금지
- hotfix는 `main` 기반으로 생성, 완료 후 `develop` 역머지

### 6. 문서와 코드를 동기화한다
- 코드 변경 시 관련 문서(DEPLOY.md, ROADMAP.md)를 함께 업데이트한다.
- 작동하지 않는 예시 코드나 잘못된 경로가 문서에 남지 않도록 한다.

### 7. 에러를 조용히 넘기지 않는다
- 예외는 명시적으로 처리하거나 상위로 전달한다.
- 로그에 충분한 컨텍스트를 남긴다.
- CI에서 실패하면 원인을 파악하고 수정한다 — `--no-verify` 우회 금지.

### 8. 보안을 기본값으로 설정한다
- API 키, DB 비밀번호 등 민감 정보는 환경 변수로 관리하고 Git에 커밋하지 않는다.
- `.env.example`에 변수 이름만 노출하고, 값은 개인 `.env`에 작성한다.
- SQL 인젝션, XSS 등 OWASP Top 10 취약점을 기본적으로 방지한다.

### 9. AI 협업 개발 원칙

이 프로젝트는 AI 에이전트가 코드를 직접 작성하는 환경이다. Karpathy의 "Software 3.0" 시대(2024-2025)에 맞는 AI 협업 원칙을 추가한다.

- **AI 생성 코드를 직접 읽고 이해한 뒤 커밋한다.** "AI가 썼으니 괜찮겠지"는 금지. 원칙 2(읽고 이해 후 수정)의 확장이다.
- **복잡한 작업은 추상화 계층 단위로 분해하여 요청한다.** 한 번에 DB 설계 + API 구현 + 프론트엔드 연동을 함께 요청하지 않는다.
- **테스트 작성을 AI에게 먼저 요청한다 (Test-First with AI).** 구현 코드보다 테스트를 먼저 생성하면 요구사항 오해를 조기에 발견할 수 있다.
- **커밋 전 `git diff`로 의도치 않은 변경을 반드시 확인한다.** AI는 맥락 파악 중 관련 없는 파일을 수정할 수 있다.
- **AI 생성 코드도 code-review 스킬의 "AI 생성 코드 리뷰 추가 체크" 항목을 통과해야 한다.**
