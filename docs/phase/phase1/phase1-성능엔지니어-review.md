> 검토 대상: Phase 1 (인프라 + 기반 도메인, Sprint 1~3)
> 검토일: 2026-05-18
> 관점: 앱 시작 3초 보장, 글로벌 검색 응답성, DB 성능, 백업 영향

## 1. 앱 시작 시퀀스 (< 3초 목표, PRD §5.6)

### 현재 설계된 시퀀스
```
앱 시작 → 동기화 대기 → 락 확인 → 무결성 검증 → 인증(비밀번호 입력) → 메인 진입
```

### 성능 분석

| 단계 | 예상 소요 시간 | 비고 |
|------|---------------|------|
| Tauri WebView 초기화 | ~500ms | 고정 비용, 최적화 불가 |
| 동기화 상태 확인 | ~100ms (정상) / 30초 (타임아웃) | **3초 예산에서 제외** — 별도 대기 화면 표시 |
| 락 파일 읽기 | ~50ms | 파일 시스템 I/O 1회 |
| 무결성 검증 | ~50ms (`quick_check`) / ~2초 (`integrity_check`) | **핵심 분기점** |
| PBKDF2 키 유도 | ~200ms (600K iterations) | CPU 바운드 |
| DB 열기 (SQLCipher) | ~100ms | |
| **합계 (정상)** | **~1000ms** | 3초 이내 충분 |

### 권고사항

| 등급 | 항목 | 설명 |
|------|------|------|
| **필수** | `PRAGMA quick_check` 사용 | `integrity_check`는 전체 DB 페이지 스캔 — 5MB에서도 ~2초. `quick_check`는 B-tree 구조만 검증하여 ~50ms. 일상 시작은 quick_check, 일일 백업 시점에 full integrity_check 실행 |
| **필수** | 시퀀스 병렬화 | 동기화 상태 확인과 락 파일 읽기를 `tokio::join!`으로 병렬 실행. 무결성 검증은 DB 열기 직후 비동기 |
| **필수** | 동기화 대기는 3초 예산 외 | 동기화 미완료 시 별도 대기 화면 표시. "동기화 대기 중" 화면은 3초 목표와 별개 |
| 권고 | PBKDF2 vs Argon2id 시간 측정 | 보안 전문가가 Argon2id 권장 — Argon2id는 PBKDF2보다 느릴 수 있음. 양 OS에서 200ms 이내 달성 가능한 파라미터 선정 |
| 권고 | Keychain 캐시 활용 | 이전 세션에서 키를 Keychain에 저장한 경우, 비밀번호 재입력 없이 Keychain에서 키 로드 → 시작 시간 단축. 단, 보안 전문가와 협의 필요 |

### 시작 시퀀스 최적화 설계
```
[Phase A: 병렬] ~100ms
  ├── 동기화 상태 확인 (미완료 시 대기 화면 분기)
  └── 락 파일 읽기

[Phase B: 인증] ~300ms
  └── 비밀번호 입력 대기 (사용자 행위 — 시간 측정 외)
  └── PBKDF2/Argon2id 키 유도 (~200ms)

[Phase C: DB 열기] ~150ms
  ├── SQLCipher DB 열기 (~100ms)
  └── PRAGMA quick_check (~50ms)

[Phase D: UI 렌더링] ~500ms
  └── Tauri WebView + React hydration

합계 (사용자 행위 제외): ~1050ms → 3초 여유 충분
```

## 2. 글로벌 검색 (< 300ms 결과 표시, PRD §4.14)

### 성능 분석
- 원생 50명 기준: 전체 목록 메모리 ~10KB
- 한글 자모 분해 + 필터링: 순수 JavaScript로 50건 ~1ms

### 권고사항

| 등급 | 항목 | 설명 |
|------|------|------|
| 권고 | 클라이언트 사이드 필터링 | TanStack Query로 원생 목록 캐싱 → 검색 시 IPC 왕복 없이 메모리 필터링. 200ms 디바운싱 + ~1ms 필터링 = 201ms (300ms 이내) |
| 권고 | 메뉴명 검색 | 메뉴명은 정적 데이터 — 하드코딩 배열에서 필터링. IPC 불필요 |
| 정보 | IPC 기반 검색 대안 | 원생 수가 500명 이상이면 백엔드 FTS5 검색 전환 검토. 50명 규모에서는 과잉 |

## 3. DB 성능 (원생 50명 CRUD < 300ms)

### 권고사항

| 등급 | 항목 | 설명 |
|------|------|------|
| **필수** | WAL 모드 | `PRAGMA journal_mode=WAL` 설정 — 읽기/쓰기 동시성 + 백업 안전성 향상 |
| **필수** | 인덱스 전략 | `students(serial_no)`, `students(name)`, `student_schedules(student_id, effective_to)` 인덱스 필수 |
| 권고 | 커넥션 풀 | sqlx Pool 크기 2~4 (단일 사용자 앱, 과도한 풀 불필요) |
| 권고 | PRAGMA 최적화 | `PRAGMA cache_size=-8000` (8MB 캐시), `PRAGMA mmap_size=268435456` (256MB mmap) — SQLCipher에서 mmap 지원 확인 필요 |

## 4. 4계층 백업 성능 영향

### 분석
- DB 5MB 기준 rusqlite::backup: ~50ms (WAL 모드, 읽기 전용)
- hourly 백업: 사용 중 1시간마다 ~50ms — 체감 불가
- exit 백업: 앱 종료 시 ~50ms — 종료 0.1초 지연

### 권고사항

| 등급 | 항목 | 설명 |
|------|------|------|
| 권고 | 백업 비동기 실행 | hourly 백업은 백그라운드 tokio task로 실행 — UI 블로킹 방지 |
| 권고 | exit 백업 동기 실행 | 종료 시 백업은 동기로 완료 후 종료 — 데이터 안전 우선 (50ms 지연 수용) |
| 정보 | 순환 삭제 | 백업 개수 초과 시 가장 오래된 파일 삭제 — 파일 시스템 I/O ~10ms |
