## 1. 프로젝트 개요
- **프로젝트 목적**: 소규모 팀이 할 일을 공유하고 진행 상태를 실시간으로 추적할 수 있는 웹 기반 협업 도구를 제공한다. 이메일·메신저에 분산된 업무 요청을 한 곳에서 관리하여 누락을 방지한다.
- **주요 사용자 및 대상 시장**: 5~20명 규모의 스타트업 및 프리랜서 팀. 별도 프로젝트 관리 도구 없이 구글 스프레드시트로 업무를 관리 중인 팀을 1차 타깃으로 한다.
- **차별화 포인트**: 설정 없이 5분 안에 팀 온보딩 완료. 복잡한 기능 없이 "할 일 생성 → 담당자 지정 → 완료" 3단계로만 이루어진 미니멀 UX.

## 2. 목표 및 성공 기준
- **핵심 목표**: MVP 출시 후 3개월 내 유료 전환 팀 50개 확보
- **성공 지표(KPI)**:
  - 월간 활성 팀 수 (MAT): 3개월 후 200팀
  - 태스크 완료율: 팀당 주간 생성 태스크의 70% 이상 완료
  - 이탈률: 가입 후 2주 내 이탈 30% 이하
- **측정 방법**: 내부 이벤트 로그(태스크 생성/완료 이벤트) + 월간 결제 전환 수 집계

## 3. 기능 요구사항

### 필수 기능 (MVP)
- ✅ 팀 워크스페이스 생성 및 초대
  - 인수 기준: 이메일 초대 링크 클릭 후 30초 안에 팀 합류 완료
- ✅ 태스크 생성·수정·삭제
  - 인수 기준: 태스크 제목, 담당자, 마감일을 설정할 수 있고 저장 즉시 팀원에게 표시
- ✅ 태스크 상태 변경 (할 일 / 진행 중 / 완료)
  - 인수 기준: 상태 변경 시 담당자에게 인앱 알림 전송
- ✅ 태스크 목록 필터링 (담당자별, 상태별)
  - 인수 기준: 필터 선택 후 200ms 이내 결과 표시

### 선택적 기능
- ⬜ 태스크 댓글 (v1.1 이후)
- ⬜ 파일 첨부 (v1.2 이후)
- ⬜ 외부 캘린더 연동 (Google Calendar) (v2.0)

### 우선순위
| 기능 | 우선순위 | 비고 |
|------|----------|------|
| 팀 워크스페이스 | High | MVP 핵심 |
| 태스크 CRUD | High | MVP 핵심 |
| 상태 변경 + 알림 | High | MVP 핵심 |
| 필터링 | Medium | MVP 포함 |
| 댓글 | Low | v1.1 |
| 파일 첨부 | Low | v1.2 |

## 4. 비기능 요구사항
- **성능**: 태스크 목록 조회 응답 시간 200ms 이하 (p95 기준, 팀당 태스크 1,000개 이하 환경)
- **보안**: 사용자 비밀번호 bcrypt 해싱, JWT 토큰 만료 24시간, HTTPS 필수
- **확장성**: 초기 100팀 동시 접속 지원, 6개월 후 1,000팀으로 수평 확장 가능한 구조
- **접근성**: 키보드 네비게이션 지원, 색상 대비 WCAG AA 기준 준수

## 5. 도메인 용어 정의
- **워크스페이스(Workspace)**: 팀이 공유하는 독립 작업 공간. 팀원과 태스크를 포함한다.
- **태스크(Task)**: 완료 가능한 단위 작업. 제목·담당자·마감일·상태를 가진다.
- **멤버(Member)**: 워크스페이스에 초대된 팀원. Owner / Editor / Viewer 권한 구분.
- **Owner**: 워크스페이스를 생성하고 멤버를 관리하는 관리자 역할.

## 6. 사용자 페르소나 (Persona)

- **Persona A — 스타트업 팀 리더 (김민준, 32세)**: 5인 스타트업의 CTO. 개발·디자인·마케팅 팀이 각자 다른 채널로 업무를 보고해 현황 파악이 어렵다. 하나의 화면에서 전체 팀 진행 상황을 보고 싶다.
- **Persona B — 프리랜서 디자이너 (이서연, 28세)**: 동시에 3개 클라이언트 프로젝트를 진행 중. 클라이언트별 요청 사항을 메모장에 적다가 누락이 발생했다. 간단한 체크리스트와 마감일 관리가 필요하다.

## 7. 사용자 시나리오 (User Stories)

- 팀 리더는 새 프로젝트 킥오프 시 워크스페이스를 만들고 팀원을 이메일로 초대할 수 있다.
- 팀원은 본인에게 할당된 태스크만 필터링하여 오늘 해야 할 일을 확인할 수 있다.
- 담당자는 태스크 완료 처리 시 팀 리더에게 자동 알림이 전송되어 별도 보고가 불필요하다.
- 팀 리더는 마감일이 지난 태스크 목록을 한눈에 확인하고 담당자에게 리마인더를 보낼 수 있다.

## 8. 데이터 요구사항
- **저장 데이터**: 사용자 계정(이메일, 비밀번호 해시), 워크스페이스, 태스크(제목·상태·담당자·마감일·생성일), 멤버십(워크스페이스 ↔ 사용자 관계)
- **처리 데이터**: JWT 세션 토큰(Redis, TTL 24시간), 인앱 알림 큐
- **보관 정책**: 완료된 태스크 1년 보관 후 아카이브. 탈퇴한 사용자 데이터 30일 내 삭제 (GDPR 준수).

## 9. 제약 조건
- **기술 스택**: React (프론트엔드) / FastAPI (백엔드) / PostgreSQL 16 (데이터베이스) / Redis 7 (세션·알림 큐)
- **운영 환경**: AWS Lightsail (단일 인스턴스), Docker Compose, GitHub Actions CI/CD
- **외부 의존성**: SendGrid (이메일 초대 발송), 없을 경우 SMTP 직접 연결로 대체 가능

## 10. 리스크 및 가정
- **가정**: 팀원은 모두 이메일 주소를 통해 초대받을 수 있다. 초기 단계에서 팀당 태스크 수는 1,000개를 초과하지 않는다.
- **리스크**: SendGrid API 장애 시 초대 이메일 발송 불가 → 재시도 큐로 대응
- **리스크**: JWT 탈취 시 계정 탈취 위험 → Refresh Token Rotation + IP 변경 시 강제 로그아웃
- **대응 계획**: 외부 서비스 의존성마다 Circuit Breaker 패턴 적용, 장애 알림 Slack 연동

> 리스크 상세 이력은 `docs/risk-register/` 참조

## 12. 규제 요건
- **GDPR**: 유럽 사용자 대상 — 계정 삭제 요청 시 30일 내 처리, 데이터 수출 기능 제공
- **개인정보보호법 (한국)**: 주민번호 수집 금지, 개인정보 처리방침 명시 의무
- **이메일 수신 동의**: 마케팅 이메일 발송 시 사전 동의 필수

## 13. 운영 및 유지보수 고려사항
- **로그 보관 기간**: 애플리케이션 로그 90일, 접근 로그 1년
- **장애 알림**: 서버 CPU 90% 초과 또는 API 에러율 5% 초과 시 Slack #alert 채널에 즉시 알림
- **배포 정책**: 무중단 배포 (Blue-Green), 롤백 절차 상세는 `docs/dev-process.md` 섹션 6.4 참조
- **DB 백업**: 매일 새벽 2시 자동 스냅샷, 7일 보관

## 14. 확장 계획
- **1단계 (MVP, 2026 Q2)**: 웹 서비스 — 워크스페이스·태스크 CRUD·알림·필터
- **2단계 (v1.1, 2026 Q3)**: 댓글·멘션, 모바일 반응형 UI 개선
- **3단계 (v2.0, 2027 Q1)**: 모바일 앱 (React Native), 외부 캘린더 연동, 공개 API 제공
