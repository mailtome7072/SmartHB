# ADR-007: 앱 잠금 인증을 6자리 숫자 PIN 으로 전환

- **상태**: Accepted
- **날짜**: 2026-05-30
- **결정자**: SmartHB 개발팀 (원장 요청 — post-Sprint 11)

## Context (배경)

앱 잠금/최초 설정 인증은 Sprint 1(T4)부터 **자유 형식 비밀번호(최소 8자)** 로 구현되어 있다. 입력한 비밀번호 → PBKDF2-HMAC-SHA256(600K iter) + salt → 32바이트 키 유도 → OS Keychain 저장(ADR-004) → 프로덕션에서 동일 키로 SQLCipher DB 암호화(ADR-001).

50대 단독 운영자(원장 1인) UX 관점에서 매 실행 시 8자 이상 영문/숫자 혼용 비밀번호 입력은 부담이라는 피드백에 따라, **6자리 숫자 PIN** 입력으로 전환을 검토한다.

### 제약 사항

- 키 유도 함수(`derive_key`)는 비밀번호를 불투명 문자열로 취급 — PIN 도 동일 경로로 동작(크립토 변경 불필요).
- 프로덕션 DB 는 클라우드 동기화 폴더(MYBOX/iCloud/Dropbox)에 위치 — DB 파일 + salt 파일이 클라우드 계정 유출 시 함께 노출 가능.
- 앱은 미출시(v0.5.0) — 기존 프로덕션 사용자 없음. 마이그레이션 부담은 개발 환경 한정.

---

## Decision (결정)

**앱 잠금 인증을 정확히 6자리 숫자 PIN 으로 전환한다.** 복구 코드(12자리, ADR 범위 밖)는 그대로 유지하여 실질 복구 강도를 보존한다.

- 프론트(`LockScreen`, `RecoveryCodeInput`): 입력을 6자리 숫자로 제한(`inputMode="numeric"`, `maxLength=6`, 숫자 외 필터), 정확히 6자리 검증.
- 백엔드(`auth::set_password`, `recovery::reset_password_with_code`): `validate_pin`(길이 6 + ASCII 숫자) 검증 추가 — UI 우회 입력 방어.
- `verify_password` 는 형식 검증 없이 키 비교만 수행(기존 유지) — 형식은 set/reset 시점에만 강제.

## Consequences (영향) — 보안 트레이드오프

### 부정적 (수용)

- **키스페이스 축소**: 6자리 숫자 = 10⁶(100만) 경우의 수. 자유 비밀번호 대비 엔트로피가 ~20비트로 급감.
- **오프라인 브루트포스 노출**: 프로덕션 SQLCipher DB 키가 PIN 에서 유도되므로, 클라우드 계정 유출로 `app.db` + `salt.bin` 확보 시 공격자가 오프라인에서 10⁶ PIN 전수 대입 가능. PBKDF2 600K iter(1회 ~0.2–0.5s)가 비용을 더하지만 GPU/병렬로 수 시간~수 일 내 돌파 가능.
- 앱 레벨 시도 횟수 제한은 오프라인 파일 공격에 무력.

### 완화책

1. **복구 코드 12자리 유지** — 키 자체의 대체 진입점은 고엔트로피로 보존.
2. PBKDF2 반복수는 현행 600K 유지(상향은 정상 잠금해제 지연과 트레이드오프 — 추후 재검토 여지).
3. 클라우드 폴더 자체의 OS 계정 보안(2FA 등)을 운영 가이드로 권고.

### 위협 모델 판단

단독 운영자·오프라인 데스크톱·가정/교습소 PC 라는 PRD 위협 모델에서, 현실적 위협은 물리적 우발 접근이며 PIN 으로 충분히 방어된다. "클라우드 동기화되는 암호화 DB"의 강도 저하는 실재하나, 미출시 단계에서 운영자 본인이 편의성과 맞바꾸기로 명시 수용한다.

## Alternatives (대안)

- **현행 유지(8자+ 비밀번호)**: 강도 우수하나 매 실행 입력 부담 — 운영자 피드백으로 기각.
- **6자리 PIN + PBKDF2 반복수 대폭 상향**: 브루트포스 비용 증가하나 정상 잠금해제도 비례 지연 — 50대 UX 와 충돌, 향후 옵션으로 보류.
- **생체인증(Touch ID/Windows Hello)**: 별도 도입 검토 대상(PRD §5.5 후속), 본 ADR 범위 밖.

## Migration (전환)

- 프로덕션: 기존 사용자 없음 — 신규 설치부터 PIN 적용.
- 개발: 기존 dev 자격증명(8자 비밀번호 기준 Keychain 키 + `salt.bin`)은 6자리 PIN 으로 매칭되지 않으므로, dev 환경은 자격증명 재설정(복구 코드 또는 Keychain 항목 + `salt.bin` 삭제 후 최초 설정 재진입) 필요. `NEXT_PUBLIC_DEV_AUTOLOGIN` 도 6자리 PIN 으로 갱신.
