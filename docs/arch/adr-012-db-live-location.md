# ADR-012: 라이브 DB 위치 — 클라우드 폴더 유지 + 접근 강화

- **상태**: Accepted (2026-07-22 사용자 확정)
- **날짜**: 2026-07-22
- **결정자**: 원장(사용자) + Claude (Sprint 23 T0)
- **관련**: `docs/incidents/2026-07-22-data-loss-rca.md` (사고 분석), ADR-009(DB 폴더 변경)

## Context (배경)

2026-07-22 프로덕션 데이터 전면 소실 사고의 근본 원인은 **라이브 SQLite/SQLCipher DB를 클라우드 동기화 폴더(MYBOX) `{cloud}/smarthb/app.db`에 열어둔 채 사용**하는 아키텍처다 (RCA 참조). 두 증상:
- **① 전면 소실**: 클라우드 파일이 유휴/강제종료로 일시 부재(dehydration)일 때 `db.rs:89 create_if_missing(true)`가 가드 없이 빈 DB를 날조 → 무결성 검사가 빈 DB를 "정상"으로 판정(`integrity.rs:311`)해 auto_restore 미발동 → 빈 상태 부팅.
- **② 유휴 후 저장 오류**: startup PRAGMA(`PRAGMA key` 등)가 커넥션이 아닌 풀에 1회만 적용(`db.rs:91-98`)되어, 유휴 중 파일 핸들 stale/교체 시 새 "맨 커넥션"이 복호화 불가 → 재시작 전까지 세션 영구 실패.

라이브 DB의 위치를 어떻게 할지 결정한다.

### 제약 (조사 결과)
- **DB 폴더 이전 기계(`change_data_folder`, copy-then-switch)는 이미 완비**(setup.rs:228) — 로컬 이전 자체는 기술적으로 가능.
- **그러나 양 PC 데이터 공유는 "단일 DB 파일이 클라우드 폴더에 있다"는 사실에만 의존**한다. 스냅샷 export/import·핸드오프 로직은 **코드에 없다**. 라이브를 로컬로 옮기면 공유 메커니즘이 사라지고 **신규 핸드오프 구축**이 필요하다.
- `backup_root = data_root/backup` 강결합(backup.rs:141) — "라이브 로컬 / 스냅샷만 클라우드" 구조는 백업 루트 분리가 필요.
- PRD: 양 PC "시점 분리 사용"(app.lock 동시 사용 차단, 순차 핸드오프) — 동시 사용은 원래 의도 아님.

### 후보
- **A안**: 클라우드 폴더 유지 + 접근 강화(가드/재연결/유휴 close). 데이터 이전 없음.
- **B안**: 로컬 라이브 DB + 클라우드 스냅샷 핸드오프(export/import 신규) + 일회성 이전.
- **C안**: 로컬 라이브 DB + 클라우드는 백업 전용(2-PC 자동 공유 포기, 2번째 PC는 수동 복원).

## 2단계: SWOT + Trade-off

**A안 (클라우드 유지 + 강화)**
- Strength: 사고의 확정 원인(빈 DB 날조·무결성 오판·키 유실·백업 축출)을 **직접 제거**. 남은 클라우드 이상은 "데이터 손실"이 아니라 **"열기 거부 + 안내"의 안전 실패**로 전환. 2-PC 모델 무손상, 이전 마이그레이션 불필요.
- Weakness: DB가 클라우드 폴더에 **잔존** → 활성 사용 중 torn-sync **잔여 리스크**(유휴 close + app.lock 단일 writer로 크게 감소, 0은 아님). 잔여가 터져도 A4/A5(복원·백업 강화)로 검증된 백업 복원.
- Opportunity: 저비용으로 즉시 실사용 안정화. B로 가는 사전 정지작업(가드·복원 강화)이 그대로 재사용됨.
- Threat: 강화가 부족하면 잔여 torn-sync가 재현될 수 있음 → 관찰 후 B 승격 트리거 필요.

**B안 (로컬 + 핸드오프)**
- Strength: 안티패턴(클라우드-라이브-DB) 자체 제거 — 이론상 가장 근본적.
- Weakness: 핸드오프 신규 서브시스템(어느 스냅샷이 최신/크래시 중 핸드오프 등 **새 실패 모드**) + 일회성 이전. 규모가 커 단일 스프린트 부적합(Phase 필요).

## Decision (결정)

**A안 채택** — 라이브 DB를 클라우드 폴더에 유지하되 접근을 강화한다. 근거: 실사용에서 발생한 두 사고(전면 소실 + 유휴 저장오류)의 **확정 원인을 근본 차단**하면서, 작동 중인 2-PC 공유 모델을 보존하고, 위험한 데이터 이전 없이 Sprint 23 안에 안전하게 담긴다. 남은 클라우드 리스크는 데이터 손실이 아닌 안전 실패/자동복원으로 전환된다.

강화 범위(Sprint 23 A 항목):
- **create_if_missing 가드**: 셋업 흔적(salt.bin) 존재 & app.db 부재 시 생성 금지 → 중단 + "동기화 대기" 안내.
- **무결성 fail-soft 제거**: 부재/빈 DB를 "정상"으로 보지 않음.
- **after_connect 훅**: 매 커넥션에 `PRAGMA key`+startup pragma 재적용(유휴 저장오류 자가회복).
- **유휴 시 연결 close + WAL 체크포인트**, 활동 시 재연결 (클라우드가 닫힌·체크포인트된 파일을 동기화하도록).
- **복원/백업 강화**: WAL 사이드카 원자 처리, 복원 소스 행수·신선도 검증, exit→daily→weekly 폴백, 빈/열세 백업 거부 + 마지막 정상 백업 축출 금지.
- **config 손상처리 통일**(무음 fallback 제거, salt.bin=SSOT) + `set_password` 기존 salt 하드 가드.

## Consequences (귀결)

- **긍정**: 오늘의 사고 시나리오 재현 불가. 유휴 저장오류 자가회복. 클라우드 이상 시 데이터 무손실(안전 실패/자동복원). 배포는 가드 추가만이라 단순.
- **부정/잔여**: 클라우드-라이브-DB 안티패턴은 남는다. 활성 사용 중 torn-sync 잔여 리스크 존재(감소됨).
- **후속(승격 트리거)**: **B안(로컬 라이브 + 클라우드 핸드오프)을 ROADMAP에 다음 Phase 후보로 명시**한다. A 배포 후에도 클라우드 간섭에 의한 손상/복원 이벤트가 관찰되면 phase-planner로 B 설계에 착수한다.
- **테스트**: 강화 로직은 인메모리 단위 테스트 + cipher 빌드 검증. "app.db 부재 시 생성 안 함", "유휴 후 재연결 시 키 재적용", "빈 백업 거부"를 회귀 테스트로 고정.
