# ADR-005: 공휴일 데이터 — API 소스 + 저장 테이블 + 갱신 주기

- **상태**: Proposed (Session #6 T2-a/b 시점에 외부 데이터 한계 발견 — 결정 자체는 유효, 데이터 범위만 축소)
- **날짜**: 2026-05-22 (초안) / 2026-05-22 (Session #6 갱신)
- **결정자**: SmartHB 개발팀 (Sprint 6 T2-c / T2-b)

## Context (배경)

PRD §4.4 학사 스케줄 캘린더는 한국 법정 공휴일을 셀에 표시해야 한다 (§4.4.1 / §4.4.4 공휴일 셀 마킹). PRD §5.5 는 앱 런타임의 외부 네트워크 호출을 금지한다 — 따라서 공휴일 데이터는 **빌드 타임에 1회 수집하여 DB 마이그레이션 시드로 박제**해야 한다.

해결해야 할 결정 2개:

1. **공휴일 데이터 API 소스**: 어디서 수집할 것인가?
2. **DB 저장 위치**: 기존 `schedule_events` 에 공휴일 code_id 참조로 INSERT 할 것인가, 별도 `holidays` 전용 테이블을 신설할 것인가?

### 제약 사항

- 7년치 (2024~2030) 데이터 수집, **대체공휴일 포함 필수** (어린이날 5/5 일요일 → 5/6 대체, 설/추석 연휴 등 — 한국 「관공서의 공휴일에 관한 규정」 시행령 반영)
- 캘린더 컴포넌트(T9, Session #6 이후)가 동일 컴포넌트 안에서 학사 일정과 공휴일을 함께 표시
- 2030년 데이터 만료 이전 갱신 필요 (사용자 1인 운영자가 직접 갱신 방법을 알 수 있어야 함)
- V102 schedule_codes 현재 시드: 보강데이 / 공휴수업일 / 방학 / 단원평가 응시일 / 휴원일 — **"공휴일" 코드 없음**. schedule_events 저장 선택 시 V301 에서 "공휴일" 코드 추가 필요
- 신규 의존성 추가는 사용자 허가 필수 — TypeScript 스크립트 + `tsx` devDependency 만 추가 (사용자 사전 승인, 2026-05-22)

## 1단계: Weighted Decision Matrix

### 1.1 API 소스 비교 — 공공데이터포털 (A) vs date.nager.at (B)

|------|--------|---------------------|--------|---------------------|--------|
| 데이터 정확도 (대체공휴일 포함) | 0.30 | 한국 천문연구원 공식 데이터, 「관공서의 공휴일에 관한 규정」 시행령 자동 반영, 대체공휴일 정확 | **5** | 한국 공휴일 부정확 보고 다수 (GitHub issues), 대체공휴일 누락 사례 (예: 2024-05-06 대체 누락 사용자 보고) | **2** |
| 운영 부담 (인증/만료/갱신) | 0.20 | 인증키 발급 1회 (회원가입 → 활용신청 → 24시간 내 발급), 보통 1년 유효, 빌드 타임 환경변수로 관리 | **3** | 무인증, 즉시 재실행 가능 — 발급/만료 부담 없음 | **5** |
| 안정성 (서비스 지속성) | 0.15 | 정부 운영 (한국정보화진흥원) — 장기 지속 보장 | **5** | 외부 무료 서비스 (1인 운영자 추정) — 단종/장애 위험 | **3** |
| 라이선스 (재배포) | 0.10 | 공공누리 제1유형 — 출처표시 후 자유 이용 가능 | **5** | MIT — 자유 이용 가능 | **5** |
| 구현 단순성 | 0.15 | XML 응답 파싱 필요 + 인증키 헤더 + 페이지네이션 | **3** | JSON 응답, GET 한 줄, 페이지네이션 없음 | **5** |
| 미래 갱신 용이성 | 0.10 | 인증키 재발급 필요 가능 (보통 자동 연장, 일부 만료 시 재발급) | **3** | 무인증 — 사용자 1인이 별도 셋업 없이 재실행 가능 | **5** |

- **B 총점** = 2×0.30 + 5×0.20 + 3×0.15 + 5×0.10 + 5×0.15 + 5×0.10 = 0.60 + 1.00 + 0.45 + 0.50 + 0.75 + 0.50 = **3.80**
- 차이 0.30 — 통계적 동등 경계선. 데이터 정확도(0.30 가중치)가 결정적 — 사용자 1인 운영자에게 잘못된 공휴일 표시는 직접적 신뢰 손상 (수업 일정 오인 → 학부모 연락 누락 등). 2단계 정성 분석에서 가중치 정당성 확인 필요.

## 2단계: SWOT + Trade-off + Risk

### 2.1 API 선택지

#### 선택지 A: 공공데이터포털 (data.go.kr 특일 정보 API)

- **Strengths**
  1. 한국 천문연구원 공식 공휴일 데이터 — 「관공서의 공휴일에 관한 규정」 시행령 변경 즉시 반영
  2. 대체공휴일법(2021 개정) 자동 반영 — 어린이날·설/추석 연휴 일요일 겹침 대체일 정확
  3. 정부 운영으로 장기 지속 보장 (서비스 단종 리스크 거의 없음)
- **Weaknesses**
  1. 인증키 발급 절차 — 회원가입 → 활용신청 → 24시간 내 발급 (사용자 1인 초기 셋업 부담)
  2. XML 응답 파싱 필요 (TypeScript 에서 fast-xml-parser 같은 추가 처리)
  3. 인증키 환경변수 관리 — `.env.example` 에 `KOREA_HOLIDAY_API_KEY` 추가 + 사용자가 본인 키 채워야 함
- **Opportunities**
  1. 향후 다른 공공 데이터(학교 정보 등) 활용 시 동일 인증키 재사용 가능
  2. 공공누리 제1유형 — 출처표시 후 별도 라이선스 비용 없이 재배포
- **Threats**
  1. API 스펙 변경 가능성 — 빌드 타임 1회 사용이므로 영향은 갱신 시점에 국한 (런타임 무관)
  2. 인증키 유효기간 만료 — 갱신 시점에 별도 발급 절차 필요

#### 선택지 B: date.nager.at

- **Strengths**
  1. 무인증 — 즉시 사용 가능, 셋업 부담 0
  2. JSON 응답 단순 — `fetch('https://date.nager.at/api/v3/PublicHolidays/2024/KR')` 한 줄
  3. 50+ 국가 지원 — 향후 다국가 확장 시 일관 API
- **Weaknesses**
  1. **한국 공휴일 부정확 보고 다수** — GitHub issues 에서 대체공휴일 누락, 2024-05-06 어린이날 대체 누락 사례 보고
  2. 데이터 소스가 GitHub PR 기반 — 정부 공식 발표 반영 지연 가능 (특히 특별 임시공휴일)
  3. 외부 무료 서비스 — 단일 운영자 추정, 단종 시 갱신 경로 끊김
- **Opportunities**
  1. 오픈소스 (MIT) — 데이터 누락 발견 시 사용자가 직접 GitHub PR 제출 가능
  2. 자체 데이터셋 다운로드 가능 (정적 JSON) — API 단종 시 fallback
- **Threats**
  1. **데이터 정확도 문제로 인한 사용자 신뢰 손상** — 한 번이라도 누락된 공휴일이 캘린더에 표시 안 되면 학부모 연락 누락 등 운영 사고 직결
  2. 운영자 단종/장애 — 갱신 시점에 가용성 보장 없음

### 2.2 저장 위치 선택지

#### 선택지 A: schedule_events 테이블에 INSERT (공휴일 code_id 참조)

- **Strengths**
  1. 캘린더 렌더 단일 쿼리 — `list_schedule_events(from, to)` IPC 가 코드 JOIN 평탄화 응답 그대로 공휴일 표시
  2. 학사 일정 코드 3속성 모델 자연스럽게 적용 — "공휴일" = 정규수업 OFF / 보강 OFF / 중복불가 ON / 기간성 OFF
  3. 지난 달 수정 차단·이동 차단 등 기존 로직 자동 적용 — 사용자가 공휴일을 실수로 수정/삭제하는 동작도 동일 가드 적용
- **Weaknesses**
  1. V102 schedule_codes 시드에 "공휴일" 코드 추가 필요 — V301 마이그레이션이 schedule_codes 시드 보정 + 공휴일 코드 INSERT + schedule_events 7년치 INSERT 3가지 작업을 함께 수행 (단일 트랜잭션)
  2. 사용자 추가 학사 일정과 공휴일이 같은 테이블 — 필터링 시 `is_system_reserved + code_name = '공휴일'` 조건 명시 필요
- **Opportunities**
  1. 향후 학교 행사·외부 일정 등 다양한 "외부 일정" 통합 시 동일 패턴 확장 가능
  2. display_name 필드 활용 — "광복절(임시공휴일)" 같은 부가 정보 표시 가능
- **Threats**
  1. schedule_events 행 수 증가 (~150) — 그러나 V103 인덱스(event_date)로 범위 조회 빠름, 미미

#### 선택지 B: 별도 holidays 전용 테이블 신설 (V302)

- **Strengths**
  1. 학사 일정과 공휴일 도메인 분리 — 개념적 깔끔함
  2. `holiday_type` (법정공휴일/대체공휴일/임시공휴일) 같은 메타필드 자유 추가 가능
- **Weaknesses**
  1. 캘린더 컴포넌트가 두 소스(`schedule_events` + `holidays`) 병합 필요 — useQuery 2개 + 합치는 로직
  2. 공휴일 전용 IPC 신규 작성 — `list_holidays`, `count_holidays` 등 추가
  3. V302 마이그레이션 + sqlx 캐시 갱신 비용
- **Opportunities**
  1. 공휴일을 학사 일정 코드 잠금에서 명시적으로 격리 — 사용자가 의도치 않게 수정할 수 없는 구조 (단, A 도 시스템 예약 코드로 동일 보호 가능)
- **Threats**
  1. 캘린더 코드 복잡도 증가 — Sprint 7 이후 §4.6 수업 캘린더 뷰에서도 동일 패턴 필요 시 부담 누적

## 3단계: Decision (결정)

> 1단계 총점: API A = 4.10, B = 3.80 (차이 0.30 — 데이터 정확도 가중치가 결정적) / 저장 A = 4.65, B = 3.10 (차이 1.55 — 결정적)

### 결정 1: API 소스 — **공공데이터포털 (선택지 A)**

**채택 근거**:
- 데이터 정확도가 본 앱의 운영 신뢰와 직결 — 잘못된 공휴일 표시는 수업·청구 일정 오인으로 학부모 연락 누락 같은 실무 사고로 이어진다. date.nager.at 의 한국 데이터 누락 보고는 이 리스크의 실증이다.
- 인증키 발급 부담은 빌드 타임 1회로 국한 — 사용자 1인이 6년에 1회(2030년 갱신) 수행 가능한 수준.
- XML 파싱 복잡도는 TypeScript + `fast-xml-parser` (또는 표준 `DOMParser`) 로 처리 가능 — 빌드 도구 코드라 본체 의존성에 영향 없음 (devDependency 만 추가).

**핵심 Trade-off**: 인증키 셋업 부담을 감수하고 데이터 정확도를 확보. 이는 PRD §5.7 "50대 사용자 친화 — 실수 복구 메커니즘" 의 정신과 일치 (사용자가 시스템 데이터 정확도에 의존해야 신뢰 형성).

### 결정 2: 저장 위치 — **`schedule_events` 테이블에 INSERT (선택지 A)**

**채택 근거**:
- 캘린더 컴포넌트(T9) 가 이미 `list_schedule_events` IPC 와 `ScheduleEventListItem` 평탄 응답으로 설계됨 (Sprint 6 T7+T8) — 공휴일을 동일 흐름에 통합하면 컴포넌트 로직 단순화.
- 학사 일정 코드 3속성 모델이 공휴일에 자연스럽게 적용 (정규수업 OFF / 보강 OFF / 중복불가 ON / 기간성 OFF) — 별도 도메인 모델 신설 불필요.
- `is_system_reserved = 1` 시스템 코드로 등록 → 기존 update_schedule_code 가드(AC-T6-1)가 그대로 공휴일 보호.

**핵심 Trade-off**: V102 코드 시드 수정 + V301 마이그레이션 작업량 증가를 감수하고 캘린더 렌더 단일성·도메인 일관성 확보. V302 별도 테이블 신설은 Sprint 7 이후 다른 외부 데이터(예: 시·도 교육청 일정) 통합 필요성이 입증될 때 재검토.

### 결정 3: 갱신 주기 + 갱신 절차 (Session #6 실측 후 갱신)

> **Session #6 발견**: data.go.kr 특일 정보 API 는 한국천문연구원 발표 시점에 따라 **1~2년치 미래 데이터만 사전 등록** 한다. 2026-05 시점 실측 결과 2028년 이후는 totalCount=0 (미발표). ADR 초안의 "7년치(2024~2030)" 가정이 외부 데이터 한계로 깨졌으나, 본 결정(API 소스 + 저장 위치)은 영향 없음 — **데이터 범위만 축소**.

- **초기 시드 범위 (V301)**: 2025~2027 (3년치, 64건) — 2024 는 과거이므로 제외 (캘린더 활용도 낮음, 사용자 결정 2026-05-22)
- **갱신 트리거 (좁힘)**: **매년 1월** (또는 천문연구원 신년 공휴일 발표 직후) `pnpm holidays:fetch -- --years YYYY-YYYY` 재실행
  - 2026-12 이전: 2028 데이터 발표 확인 후 V401 추가 권장
  - 매년: N년치 신규 데이터 발표 시 즉시 갱신
- **갱신 절차** (사용자 1인 운영자 기준):
  1. data.go.kr 인증키 유효성 재확인 (보통 자동 연장, 만료 시 재발급)
  2. `pnpm holidays:fetch -- --years <시작>-<종료>` 실행 → stdout SQL INSERT 문 → `scripts/holidays.generated.sql` 저장
  3. 다음 사용 가능 마이그레이션 번호로 신규 파일 생성 (예: V401, V402…)
  4. 생성된 SQL을 V301 의 "3단계" 패턴 (CROSS JOIN VALUES + `WHERE c.code_name='공휴일' AND c.is_system_reserved=1`) 형식으로 박제
  5. academic.rs 에 신규 연도 주요 공휴일 검증 테스트 추가 + `cargo test`
  6. `git commit` (사용자 1인 직접 머지)
- **트리거 메모**: ROADMAP.md 에 "공휴일 시드 갱신: 매년 1월" 메모 (sprint-close 에이전트가 Sprint 6 회고에 반영)
- **갱신 누락 영향**: 미갱신 연도는 캘린더에 공휴일 표시 안 됨 — 사용자가 시각적으로 즉시 인지 가능 (12월 25일에 배지 없음 등). 운영 사고 위험 낮음.

## Consequences (영향)

**긍정적 영향**:
- 캘린더 컴포넌트(T9)가 공휴일·교습기간·학사 일정을 단일 데이터 흐름으로 렌더 — 코드 복잡도 최소화
- 시스템 예약 코드 가드(`is_system_reserved = 1`)로 사용자 실수 자동 방지 — 학사 일정 보호 메커니즘 재사용
- 공휴일 데이터 정확도 확보로 운영 신뢰 형성 — 50대 사용자의 시스템 의존성 강화
