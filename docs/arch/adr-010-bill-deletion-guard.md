# ADR-010: 청구 삭제 가드 정책

- **상태**: 채택 (Accepted)
- **일자**: 2026-07-19
- **관련 스프린트**: Sprint 20 (T2/T3/T4)
- **결정권자**: 원장(사용자) — 2026-07-19 확정

## 배경 (Context)

실사용 중, 잘못 생성된 청구(예: 교습기간 종료 이후 입교한 원생의 과청구, T1로 근본 원인은 수정됨)를
정정할 수단이 없다는 문제가 드러났다. 기존 청구 도메인에는 **삭제 커맨드(`delete_bill`)가 전무**했고,
유일한 정정 수단은 `update_bill`로 조정금액을 0으로 만드는 것뿐이었다(청구서 자체는 목록에 잔존).

`payments.bill_id`는 `ON DELETE CASCADE`(V109)이므로 `bills` 행을 삭제하면 연결된 수납 행도 자동 삭제된다.
따라서 기술적 삭제는 가능하나, **어떤 상태의 청구까지 삭제를 허용할지(가드 정책)**를 정해야 한다.

시스템은 "잘못된 청구"를 자동 판별하지 않는다 — 원장이 청구 화면에서 대상을 직접 선택해 삭제한다.
따라서 가드는 **청구 상태(미확정/확정, 미수납/수납완료)** 기준으로 삭제 허용 범위를 제한한다.

## 검토 대안 (Options)

| 안 | 삭제 허용 범위 | 장점 | 단점 |
|----|---------------|------|------|
| **A** | 미확정(draft) + 미수납만 | 가장 보수적, 실수 여지 최소 | 이미 확정된 오류 청구를 정정 불가 |
| **B** ✅ | 미수납이면 draft/confirmed 무관 삭제 / 수납완료 거부 | 정정 유연성 + 수금 완료 데이터 보호 균형 | 확정 청구 삭제 시 주의 필요(다이얼로그로 보완) |
| **C** | 모든 상태(수납완료 포함) 삭제 | 최대 유연성 | 수금 완료 데이터까지 실수 삭제 위험 |

### 평가 기준별 비교 (가중)

| 기준 | 가중 | A | B | C |
|------|------|---|---|---|
| 데이터 안전성 (수금 완료 보호) | 0.35 | 5 | 5 | 2 |
| 정정 유연성 (1인 원장 실무) | 0.30 | 2 | 5 | 5 |
| 실수 방지 | 0.20 | 5 | 4 | 2 |
| 구현 복잡도(낮을수록 高) | 0.15 | 5 | 4 | 4 |
| **가중 합** | | **4.10** | **4.65** | **3.10** |

## 결정 (Decision) — **B안 채택**

- **미수납(`payments.is_paid = 0` 또는 payments 행 없음)**: 상태가 `draft`든 `confirmed`든 **삭제 허용**.
- **수납완료(`is_paid = 1`)**: **삭제 거부**. 먼저 수납을 해제(`is_paid=0`)한 뒤 삭제해야 한다.
- `confirmed`(확정) 청구 삭제 시, 프론트 확인 다이얼로그에 확정 상태임을 명시(위험 동작 경고, `frontend.md` 실수 복구 규칙).
- 삭제 시 감사 로그(`AuditEventType::BillDeleted`) 기록 — 대상 원생/청구월/금액/상태/수납동반 여부.

### 기각 사유

- **A안 기각**: 원장이 오류 청구를 이미 확정(검토 완료)했을 때 정정할 수 없어, 이번 이슈의 실제 정정 목적을 달성하지 못한다.
- **C안 기각**: 수금이 끝난(is_paid=1) 청구까지 한 번의 클릭으로 삭제 가능하면, 실수로 정산 완료 데이터를 잃을 위험이 크다. 수납 해제라는 명시적 1단계를 거치게 하여 안전장치를 둔다.

## 결과 (Consequences)

- `delete_bill` IPC는 삭제 전 `bills LEFT JOIN payments`로 `is_paid`를 확인하여 `is_paid=1`이면 거부한다(`update_bill` 291행 패턴 재사용).
- `payments`는 CASCADE로 함께 삭제되므로 수납 목록(`list_payment_view`)에서도 자동 제거 — 별도 수납 IPC 수정 불필요.
- 실 DB 보정(T5) 시, 대상 청구가 수납완료 상태면 **① 수납 해제 → ② 삭제** 순서로 진행한다.
- V111에서 폐기된 '마감(closed)' 개념은 고려하지 않는다(청구 상태는 draft/confirmed 2단계).
