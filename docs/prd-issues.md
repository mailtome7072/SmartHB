# PRD v1.5 정합화 이슈 리포트

> **목적**: PRD.md v1.4~v1.5 정합화 과정에서 발견된 **논리적 오류, 맥락 단절, 정의의 어긋남**을 식별하여 사용자 검토 후 PRD 패치 여부를 결정한다.
> **작성일**: 2026-05-15 (초기) / 2026-05-18 (v1.5 정합화 후 갱신)
> **PRD 수정 권한**: 사용자 명시적 허가 후 별도 작업으로 PRD에 반영. 본 문서 작성만으로 PRD를 수정하지 않는다.
> **결정 방법**: 각 이슈의 "결정" 컬럼을 `PRD 패치` / `후순위` / `기각` 중 하나로 채운다.
>
> **v1.5 처리 현황**:
> - 학습보고서 분기 단위 재설계 → PRD v1.5에서 §4.8 전면 개정
> - 청구 마감 3단계 → PRD v1.4 §4.9.7에서 신설 (PI-04 일부 해소)
> - PI-11 E2E 도구 → **결정: PRD 패치** (PRD §6.5를 `Tauri WebDriver(tauri-driver)`로 통일)
> - 학습보고서 출력 범위 모순 (§4.8.4 vs §4.13.2) → **결정: PRD 패치** (§4.8.4에서 "파일 저장 없음" 제거)

## 상세 이슈

### PI-01

- **위치**: PRD §4.5.7 (소멸 규칙) vs AC-4.5-1 (출결 생성은 동일 월 1회만)
- **카테고리**: 시간 의존성
- **심각도**: High
- **현 PRD 인용**:
  > §4.5.7: "다음 월 출결 생성 시 소멸기한 도래 + 미보강 결석은 '보강소멸' 상태로 자동 전이"
  > AC-4.5-1: "출결 생성은 동일 월에 1회만 가능, 재실행은 v2.0 이후 검토"
- **문제**: 다음 월 교습기간이 미수립 상태이면 다음 월 출결 생성이 불가하므로, 소멸기한이 실제 도래해도 자동 전이 트리거가 발동하지 않는다. §4.11.4 "다음 달 학사 미수립 경고"가 트리거되는 시점부터 자동 전이 시점까지 공백이 존재.
- **수정 권장안**: 별도 batch 로직 추가 — "앱 시작 시" 또는 "출결 화면 진입 시" 또는 "교습기간 등록 직후" 중 하나의 트리거에서 미보강 결석 중 소멸기한 도래한 항목을 일괄 "보강소멸" 전이 처리한다.
- **결정**: (사용자 검토)

### PI-02

- **위치**: PRD §6.1 도메인 개념도 vs §4.5.4 보강 등록 (개별)
- **카테고리**: 데이터 모델
- **심각도**: High
- **현 PRD 인용**:
  > §4.5.4: "매칭 단위는 일(day) 단위 — 결석 1일과 보강 1일이 1:1로 대응"
  > §4.5.4: "보강 1건이 결석 N건 충당 가능 (예: 보강데이 하루로 결석 2일치 처리)"
  > §6.1: "보강-결석 매칭 (일 단위 1:N, 보강 1건이 결석 N건 충당 가능)"
- **문제**: "1:1 대응"과 "1:N 충당"이 동시에 성립하려면 결석/보강의 **시간값** 처리 규칙이 명확해야 한다. 보강 1일이 4시간이면 결석 4건(각 1시간)을 충당하는가, 결석 2건(각 2시간)을 충당하는가? 시간값 합산 vs 일자 수 합산 기준이 모호하다.
- **수정 권장안**: "보강 1건의 수업시간 ≥ 충당하려는 결석들의 수업시간 합" 제약을 명시하거나, 또는 "일 단위 매칭은 시간 무관 — 보강 1일이 결석 N일을 충당하되 잔여 시간은 폐기"로 명확화. §6.1 도메인 개념도에도 시간값 처리 규칙 추가.
- **결정**: (사용자 검토)

### PI-04

- **위치**: PRD §4.9.1 청구 데이터 생성 vs §4.9.2 월 중 입퇴교 처리
- **카테고리**: 상태 흐름
- **심각도**: Medium
- **현 PRD 인용**:
  > §4.9.1: "청구 항목: 성명, 청구년월, 주단위 총 수업시간, 청구 교습비, 확정 여부, 월 중 입퇴교 플래그"
  > §4.9.3: "청구 데이터는 '미확정' 상태로 생성됨"
  > §4.9.2: "월 중 입교·퇴교 원생도 표준 청구액으로 일괄 생성 ... 원장이 해당 원생의 청구액을 수동 조정 후 확정 처리"
- **문제**: 모든 청구가 "미확정"으로 생성되는데, §4.9.2는 "수동 조정 → 확정"으로 직접 전이하는 것처럼 읽힌다. 검토 단계(미확정 유지 → 검토 완료 → 확정)가 표현상 누락.
- **수정 권장안**: §4.9.2를 "원장이 해당 원생의 청구액을 수동 조정한다. 수동 조정 후에도 '미확정' 상태를 유지하며, 검토 완료 시 §4.9.3 절차로 '확정' 처리한다." 로 명확화.
- **결정**: ✅ **PRD 패치 부분 반영 (v1.4)** — PRD v1.4 §4.9.7에서 청구 마감 3단계(미확정→확정→마감) 워크플로우 신설로 상태 전이 명확화. 다만 §4.9.2 본문은 미수정 — 추후 §4.9.7과 cross-reference 추가 권장.

### PI-08

- **위치**: PRD §5.4 자동 백업 정책 + §5.1 비용
- **카테고리**: 비용 가정
- **심각도**: Medium
- **현 PRD 인용**:
  > §5.4: 백업 4계층 — exit(10) + hourly(24) + daily(30) + weekly(4) = 총 68개 파일
  > §5.1: "월 0원 (네이버 MYBOX 무료 30GB 한도 내 운영)"
- **문제**: DB 파일이 매시간 + 매 종료 + 매일 + 매주 시점에 백업되어 OS 클라우드 동기화 폴더에 누적된다. 평균 DB 크기를 가정하지 않으면 30GB 한도 안에 들어가는지 검증 불가. 또한 매시간 백업 파일이 클라우드 동기화 트래픽을 발생시켜 양 PC 동기화 지연을 유발할 수 있다.
- **수정 권장안**:
  - 평균 DB 크기 가정 추가 (예: 원생 50명 기준 ~5MB → 백업 68개 × 5MB = 340MB → 무료 한도 안전)
  - 비정상 증가(이상치) 감지 정책 추가 — DB 크기가 N배 증가 시 hourly 보관 개수를 줄이거나 사용자 알림
  - 백업 파일을 동기화 폴더가 아닌 로컬 별도 위치에 저장하는 옵션 검토 (단, 양 PC 간 백업 공유 불가)
- **결정**: (사용자 검토)

### PI-09

- **위치**: PRD §5.3 동시성 제어
- **카테고리**: 경계조건
- **심각도**: Medium
- **현 PRD 인용**:
  > §5.3: "락 존재 + 다른 디바이스 점유 중 → '강제 점유' (5분 이상 갱신 없는 락만 가능 — 비정상 종료 케이스 복구용)"
  > §5.3: "사용 중 매 60초마다 락 타임스탬프 갱신"
- **문제**: 정상 사용 중에도 클라우드 동기화 지연(MYBOX 클라이언트 일시 장애, 네트워크 변동 등)으로 락 파일이 5분 이상 미동기화될 수 있다. 이 경우 상대 PC에서 "강제 점유"가 발동되어 양쪽이 동시에 점유하는 상황(write 충돌) 위험.
- **수정 권장안**:
  - 강제 점유 시 사용자에게 명시적 경고: "다른 PC에서 동기화 지연 중일 수 있습니다. 강제 점유 후 데이터 충돌 시 자동 백업에서 복구 가능합니다."
  - 강제 점유 후 첫 트랜잭션 직전 DB 무결성 검증 자동 실행
  - 5분 임계 시간을 사용자 설정 가능 옵션으로 (기본 5분, 클라우드 동기화 지연 잦은 환경은 10분으로 조정)
- **결정**: (사용자 검토)

- **카테고리**: 단절

- **현 PRD 인용**:

  > §5.3: "DB 단일 파일(`app.db`)을 양 PC가 공유하는 클라우드 동기화 폴더에 둠"

### PI-11

- **위치**: PRD §6.5 테스트 전략
- **카테고리**: 테스트 전략
- **심각도**: Medium
- **현 PRD 인용**:
  > §6.5: "E2E: 핵심 사용자 흐름(UC-1 ~ UC-6) Playwright"
- **문제**: 일반 Playwright는 웹 브라우저(Chromium/Firefox/WebKit) 대상이며 Tauri 데스크톱 앱의 WebView를 직접 제어할 수 없다. Tauri E2E는 `tauri-driver` (WebDriver 기반) 또는 `playwright-webview` 같은 별도 도구가 필요.
- **수정 권장안**: §6.5의 "Playwright"를 다음 중 하나로 구체화.
  - (A) "Tauri WebDriver (`tauri-driver`) 기반 E2E" — 공식 Tauri 권장
  - (B) "Next.js dev server 모드에서 Playwright 사용 (Tauri 셸 외부에서 UI 흐름만 검증)" — 데스크톱 통합 테스트 불가
  - (C) 둘을 병행하여 (A)는 핵심 UC, (B)는 일반 UI 회귀
- **결정**: ✅ **PRD 패치** (2026-05-18) — (A) `Tauri WebDriver(tauri-driver)`로 통일. PRD §6.5 본문 갱신 완료.
