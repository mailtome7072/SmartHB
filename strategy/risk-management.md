# 리스크 관리 지침

> **역할**: 프로젝트 리스크 식별, 평가, 대응 원칙을 정의한다.
> **산출물**: `docs/risk-register/` 폴더에 리스크 이력을 기록한다.

---

## 리스크 분류

| 영향도 | 기준 |
|--------|------|
| 높음 | 프로덕션 장애, 데이터 손실, 보안 침해 |
| 중간 | 기능 저하, 일정 지연 1주 이상 |
| 낮음 | 사소한 버그, 일정 지연 1주 미만 |

## 리스크 대응 우선순위

1. **높음**: 즉시 Hotfix 프로세스 또는 긴급 조치 수행
2. **중간**: 다음 스프린트에 포함하거나 Sprint로 대응
3. **낮음**: 백로그에 기록, 우선순위 조정

---

## 기록 방식: 자동 vs 수동

### 자동 기록 (agent가 처리)

| 트리거 | 담당 agent | 기록 내용 |
|--------|-----------|---------|
| 스프린트 계획 수립 시 예상 리스크 | `sprint-planner` | ROADMAP 분석 중 식별한 블로커, 의존성 위험, 불확실성 |
| 코드 리뷰에서 Medium/High 이슈 발견 시 | `sprint-review` | 보안·성능·품질 이슈 중 영향도 중간 이상 항목 (파일 없으면 신규 생성, 있으면 추가) |

### 수동 기록 (개발자가 직접 작성)

아래 상황은 사람이 직접 판단하여 `docs/risk-register/`에 기록한다:

- 외부 API/서드파티 라이브러리 변경 감지
- 팀 인력 변동, 일정 압박 등 프로젝트 환경 변화
- 스프린트 진행 중 발견한 기술적 부채 중 즉시 대응이 어려운 항목
- hotfix 발생 원인이 된 근본 리스크

**수동 기록 시점**: 리스크 인지 즉시 → `docs/risk-register/YYYY-MM-DD.md`에 추가

---

## 산출물 형식

```markdown
# Risk Register - YYYY-MM-DD

| ID | 설명 | 영향도 | 출처 | 대응 계획 |
|----|------|--------|------|-----------|
| R1 | API 변경 가능성 | 높음 | 수동 | 버전 고정 및 모니터링 |
| R2 | 인증 로직 복잡도 | 중간 | sprint-close 코드 리뷰 | 다음 스프린트에서 리팩토링 |
```

> `출처` 컬럼: `sprint-planner` / `sprint-close` / `수동` 중 하나로 기재

## Semantic Versioning 기준

> 이 기준은 리스크 규모 평가(Major 변경의 영향도)와 연동되어 여기에 정의합니다. `CHANGELOG.md` 버전 번호 및 `deploy-prod` 에이전트의 배포 버전 결정 시 아래 기준을 적용합니다.

| 버전 올림 | 조건 | 예시 |
|-----------|------|------|
| **Major** (x.0.0) | 하위 호환이 깨지는 API 변경, 인증 방식 교체, DB 마이그레이션 필수 | 1.0.0 → 2.0.0 |
| **Minor** (x.y.0) | 하위 호환을 유지하는 새 기능 추가, 새 API 엔드포인트 | 1.2.0 → 1.3.0 |
| **Patch** (x.y.z) | 버그 수정, 핫픽스, 비기능 개선 (성능, 로그) | 1.2.3 → 1.2.4 |

> 첫 프로덕션 배포 전까지는 `0.y.z` 표기를 사용합니다. (예: 0.1.0)
